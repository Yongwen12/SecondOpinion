# 验证策略与并行工作计划

本文是 SecondOpinion 的内部产品与验证策略说明。它补充 [标注手册](./标注手册.md)，但不面向标注者公开。

## 1. A/B 不是先标注再决定

Claim 的两个核心 construct 要先在产品上分清：

- Construct A: claim epistemic quality。问题是“这条 reviewer criticism 本身站不站得住”。这是规范性判断，需要人类专家给 gold。
- Construct B: claim lifecycle influence。问题是“这条 concern 是否在真实 review lifecycle 中传播、影响 AC/meta-review/decision”。这是描述性信号，系统可以从 OpenReview lifecycle 数据里计算。

标注不会替产品选择 A 还是 B。正确顺序是：

1. 先凭产品判断确定主目标。
2. 再用 pilot 验证这个目标是否可被可靠测量和交付。
3. 最后根据 human-human agreement、system-human agreement 和 A/B 相关性决定是否扩标。

对当前项目，建议方向是：

- 以 Construct B 作为近期可验证核心：作者侧 rebuttal triage，即“哪些 reviewer concerns 更可能影响 AC 或 meta-review”。
- 以 Construct A 作为长期愿景：reviewer/venue quality audit，即“这条 criticism 是否真的高质量”。
- 前端可以双轴展示，但优化目标和对外主叙事要先押 B。

原因：B 的 ground truth 大部分来自已有公开 lifecycle 数据，验证成本低；A 必须依赖专家 gold，并且先要证明专家之间能稳定一致。

## 2. Pilot 的真正作用

IAA pilot 不是用来决定要不要做 A 或 B，而是用来回答：

- 人类专家能否稳定标注 Construct A？
- `claim_quality` 是否有足够高的一致性，可以当 gold？
- 人类 `claim_quality` 和系统 `lifecycle_robustness` 是否相关？
- 系统的 `rebuttal_resolution` 和 `recommended_action` 是否接近专家判断？

如果 `claim_quality` 的 Krippendorff alpha 足够高，A 可以进入 full expert labeling。

如果 alpha 很低，说明“holistic review quality”这个 construct 过软，需要退回到更窄的可一致维度，例如 `grounding_accuracy`、`rebuttal_resolution` 或 `recommended_action`。

## 3. 当前推荐定位

近期 investor/demo 叙事建议：

> SecondOpinion helps authors triage reviewer claims by estimating which concerns are evidence-grounded, unresolved, and likely to matter in the review lifecycle.

中文版本：

> SecondOpinion 帮作者把 reviewer comments 拆成 claim，并判断哪些 concern 有证据、有生命周期影响、需要优先 rebuttal。

暂时避免主打：

> We can objectively score reviewer quality.

这个说法要等 Construct A 的 expert IAA 和 gold validation 通过后再说。

## 4. 可以并行推进的工作

### Tier 1: 不依赖专家标注，优先做

#### 4.1 拆分并量化 lexical proxy 风险

目标：验证 concern-survival、inter-reviewer consensus、rebuttal-resolution 里由 lexical overlap 带来的假信号有多大。

具体工作：

- 增加 ablation 开关，关掉或弱化 lexical matching。
- 分别报告 lexical-only、LLM-calibrated、hybrid 三组结果。
- 重测“rebuttal 词面响应多，但真正解决少”这个结论是否仍成立。
- 重测 RAG/case-memory 信号是否仍成立。

这件事最重要，因为它直接影响项目最亮的 empirical finding 是否站得住。

#### 4.2 用已有 lifecycle 数据做 B 的 outcome validation

目标：不花专家标注钱，先验证系统分数是否能预测真实 lifecycle outcome。

候选 outcome：

- concern 是否被 meta-review 采纳。
- concern 是否在 post-rebuttal discussion 中继续出现。
- reviewer concern 是否被 author rebuttal resolved 或 weakened。
- review-level reliability 是否与后续 AC/meta-review 行为相关。

建议做法：

- 按 paper 划分 train/validation，避免同一 paper 泄漏。
- 在 train 上确定阈值或权重。
- 在 held-out papers 上报告 Spearman correlation、AUC 或 top-k precision。
- 明确区分 prediction target：不是 paper acceptance，而是 reviewer-claim lifecycle influence。

#### 4.3 接现成标注数据集或外部 gold

目标：减少自建专家标注量，用已有数据先校验 claim extraction、grounding 和 rebuttal-action schema。

候选方向：

- SubstanReview 类 claim-evidence 标注数据。
- ReviewCritique 类 deficiency / rebuttal-action 标注数据。
- 公开 peer-review assessment 数据集。

需要做的是 schema mapping，不是马上训练模型。

### Tier 2: 便宜但能降低未来风险

#### 4.4 写清 construct 定义和临时产品方向

标注手册已经定义了 A/B。下一步可以在 README 或 progress report 里把“近期主目标是 B，A 是后续专家验证目标”写清楚，避免团队和投资人误解。

#### 4.5 加隐私和保密边界

需要明确区分：

- 公开 OpenReview 历史数据：可用于研究和 demo。
- 在审稿件、未公开 reviewer text、private submission：需要强隐私边界，不能默认上传或外发。

未来若做作者侧实时助手，必须有：

- local/private processing 选项。
- prompt injection 防护。
- 对 paper/review text 的数据保留策略。

### Tier 3: 沟通和 demo

#### 4.6 做双轴 claim card

前端可以同时显示：

- Quality estimate: A 相关，先标为 expert-pending。
- Lifecycle influence: B 相关，系统当前可计算。

示例解释：

- High quality / high influence: 必须重视。
- High quality / low influence: 可能是 AC 低估了的有效批评。
- Low quality / high influence: 需要小心 rebuttal，因为它可能影响 AC，但要用证据纠正。
- Low quality / low influence: 低优先级。

这有助于 demo，但不能替代验证。

## 5. 暂缓事项

以下事项先不要投入太多：

- SFT / preference data 大规模导出。
- 新增第二个昂贵 external provider。
- 直接扩到 200-300 篇专家标注。
- 用 lifecycle robustness 直接当 review quality gold。

这些都依赖前面 Tier 1 和 IAA pilot 的结果。

## 6. 推荐下一步

最短路径：

1. 把当前产品主目标临时定为 Construct B: author-facing lifecycle triage。
2. 立刻做 lexical proxy ablation。
3. 并行做 held-out lifecycle outcome validation。
4. 准备 IAA calibration round，先 2-3 篇、3 位 annotator。
5. 用 pilot 结果决定是否扩展 Construct A 的 expert gold。

如果本周五要对投资人汇报，建议讲：

> We already have a working lifecycle-based reviewer-claim triage system. The next validation step is not blind scale; it is a cheap IAA pilot that tests whether human experts agree on normative claim quality, while we continue validating lifecycle influence using existing OpenReview outcomes.
